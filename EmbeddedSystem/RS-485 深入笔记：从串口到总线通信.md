## 1. 引言
我们知道串口是一种通信"协议框架"，"串口"（Serial Port / UART）全称是 **[[UART(通用异步收发器)|通用异步收发传输器]]**（Universal Asynchronous Receiver-Transmitter），它定义的是**数据如何打包、发送时序**，但是UART 只管"格式"，不管"电压"，**UART 协议本身不规定用什么电压传输**，  电压标准由物理层决定，常见的有：
- TTL 电平（3.3V / 5V 逻辑）
- RS-232 电平（±12V）
- RS-485 差分电平

## 2. TTL 电平 vs RS-232 vs RS-485

| 特性      | TTL（3.3V）   | RS-232        | RS-485           |
| ------- | ----------- | ------------- | ---------------- |
| 逻辑"1"电压 | 2.0V ~ 3.3V | -3V ~ -15V    | VA - VB > +200mV |
| 逻辑"0"电压 | 0V ~ 0.8V   | +3V ~ +15V    | VA - VB < -200mV |
| 传输线数量   | 2根（TX/RX）   | 3根（TX/RX/GND） | 2根（A/B，差分）       |
| 最大传输距离  | ~1米         | ~15米          | **1200米**        |
| 最大传输速率  | 一般<1Mbps    | 115200bps常用   | **10Mbps（短距）**   |
| 抗干扰能力   | 弱           | 中             | **强**            |
| 多机通信    | 不支持         | 不支持           | **支持（最多32节点）**   |

### 2.1 TTL 直连（电池1，UART3）
```
STM32 UART3_TX (PA10) ──────────────────────── 智深电池 RX
STM32 UART3_RX (PA9)  ──────────────────────── 智深电池 TX
GND ──────────────────────────────────────── GND
逻辑"1" = 3.3V
逻辑"0" = 0V
```
**适用场景**：两个设备近距离（板上/同机箱），电平相同，直接连线即可。  
**问题**：超过1米信号就可能失真；只能1对1，不能总线挂多个设备；抗干扰差。
## 3. RS-485 核心原理：差分信号
### 3.1 什么是差分信号
普通TTL是**单端信号**：用一根线的电压相对GND来判断0或1。  
RS-485是**差分信号**：用**两根线之间的电压差**来判断0或1。
关键：   A - B > +200mV → 逻辑"1"（Mark）
        A - B < -200mV → 逻辑"0"（Space）

### 3.2 为什么差分信号抗干扰强
干扰（噪声）会**同时**叠加到A线和B线上（共模噪声），在计算 A-B 时，噪声被抵消了：
```
正常信号：    A = 3.5V,  B = 0.5V   → A-B = +3.0V  → 逻辑"1" ✓

有干扰时：    A = 3.5V + 1.0V(噪声) = 4.5V
              B = 0.5V + 1.0V(噪声) = 1.5V
              A-B = 4.5 - 1.5 = +3.0V          → 逻辑"1" ✓ （噪声被抵消！）
```
这就是为什么RS-485能在工业环境中走1200米——因为干扰对两根线影响相同，不影响差值。
### 3.3 RS-485 的线命名
| 线名  | 别名   | 说明             |
| --- | ---- | -------------- |
| A   | D+、+ | 正端，发送"1"时，A比B高 |
| B   | D-、- | 负端，发送"1"时，B比A低 |

> **注意**：不同厂商命名可能相反！接线前务必查对方设备手册。有的厂商把高电平端叫B，低电平端叫A。判断方法：用万用表量，总线空闲时哪根高就是A（D+）。


## 4. 半双工 vs 全双工
### 4.1 全双工（TTL UART，电池1）
```
设备A TX ──────────────────► 设备B RX
设备A RX ◄────────────────── 设备B TX
```
两条独立的单向通道，可以**同时收发**，互不干扰。
### 4.2 半双工（标准RS-485）
```
设备A ◄──────── A/B总线 ────────► 设备B
         ↑                    ↑
       某时刻只有一个设备在发送
```
RS-485 通常只用**2根线**（A、B），同一时刻只能有一个方向的数据流动：
- **发送时**：把 TX 信号驱动到总线上，此时不能接收
- **接收时**：监听总线，此时不能发送
这要求软件层必须实现**收发切换**，这就是 DIR 引脚存在的原因。

> **4线RS-485（全双工）**：也存在4线RS-485方案（独立的发送对TX+/TX- 和接收对RX+/RX-），但在嵌入式中很少见，成本高，本项目用的是标准2线半双工。
## 5. DIR 引脚：收发切换的关键

### 5.1 为什么需要 DIR 引脚
MAX485 芯片有两个使能控制引脚：
- **DE（Driver Enable）**：高电平时允许发送（驱动总线）
- **/RE（Receiver Enable）**：低电平时允许接收（监听总线）
通常将 DE 和 /RE 连在一起，用 STM32 的一个 GPIO 来控制，这就是 **DIR 引脚**：
```
DIR = 1（高电平）：发送模式
    DE = 1 → 发送使能 ON
   /RE = 1 → 接收使能 OFF

DIR = 0（低电平）：接收模式  
    DE = 0 → 发送使能 OFF
   /RE = 0 → 接收使能 ON
```
### 5.2 DIR 引脚时序
```
正确的发送时序：
1. DIR = 1           ← 切换到发送模式（先拉高）
2. 等待几微秒（可选，让总线稳定）
3. 发送 UART 数据
4. 等待发送完成（等UART TX FIFO清空）
5. DIR = 0           ← 切换回接收模式（必须在发送完成后！）
时序图：
DIR  ___________
    |            |____________
    |发送使能     |接收使能
DATA       |帧1|帧2|帧3|
              ↑             ↑
         发送开始       发送结束后立即拉低
```
**常见错误**：发送数据还没到总线上（UART还在发），就把DIR拉低了，导致最后几个字节被截断！
### 5.3 如何确认发送完成
在 STM32 HAL 库中，有几种方式：
```c
/* 方法1：轮询发送，HAL_UART_Transmit 返回后，数据已经完全发出 */
HAL_GPIO_WritePin(USART2_DIR_GPIO_Port, USART2_DIR_Pin, GPIO_PIN_SET);   // DIR=1 发送
HAL_UART_Transmit(&huart2, buf, len, 100);                                // 阻塞直到发完
HAL_GPIO_WritePin(USART2_DIR_GPIO_Port, USART2_DIR_Pin, GPIO_PIN_RESET); // DIR=0 接收

/* 方法2：DMA/中断发送，需要在发送完成回调里拉低 DIR */
void HAL_UART_TxCpltCallback(UART_HandleTypeDef *huart) {
    if (huart->Instance == USART2) {
        HAL_GPIO_WritePin(USART2_DIR_GPIO_Port, USART2_DIR_Pin, GPIO_PIN_RESET);
    }
}
```
## 6. MAX485 芯片详解
MAX485 是最常用的 RS-485 收发器芯片，以下是其引脚说明：
```
            MAX485 引脚图
          ┌──────────────┐
  RO  1 ──┤              ├── 8  VCC (3.3V 或 5V)
  /RE 2 ──┤   MAX485     ├── 7  B (差分负端)
  DE  3 ──┤              ├── 6  A (差分正端)
  DI  4 ──┤              ├── 5  GND
          └──────────────┘
```

| 引脚  | 名称              | 方向        | 说明             |
| --- | --------------- | --------- | -------------- |
| RO  | Receiver Output | 输出→MCU RX | 接收到的串行数据输出给单片机 |
| /RE | Receiver Enable | 输入←MCU    | 低电平有效，使能接收器    |
| DE  | Driver Enable   | 输入←MCU    | 高电平有效，使能发送驱动器  |
| DI  | Driver Input    | 输入←MCU TX | 单片机发送的串行数据     |
| A   | 差分正端            | 接总线       | 总线A线（高电平端）     |
| B   | 差分负端            | 接总线       | 总线B线（低电平端）     |
| VCC | 电源              | —         | 3.3V 或 5V（查型号） |
| GND | 地               | —         | 共地             |
### 6.1 完整连接图
```

STM32                    MAX485              RS-485 总线
                       ┌──────────┐
PA2 (USART2_TX) ──────►DI         │
PA3 (USART2_RX) ◄──────RO         │
PA1 (DIR_GPIO)  ──┬───►DE        A├──────── A线 ──────► 电池2
                  └───►/RE(取反)  B├──────── B线 ──────► 电池2
                       │         │
                3.3V──►VCC      GND◄── GND
                       └──────────┘
```
> **注意**：DE 和 /RE 虽然逻辑相反，但通常直接连到同一个 GPIO（因为 /RE 低有效、DE 高有效，刚好互补，一个引脚就能控制）。
### 6.2 MAX485 同类芯片对比

| 芯片       | 供电      | 速率      | 特点                 |
| -------- | ------- | ------- | ------------------ |
| MAX485   | 5V      | 2.5Mbps | 经典型号，需外部 DE/RE 控制  |
| MAX3485  | 3.3V    | 10Mbps  | 适合3.3V系统           |
| SP3485   | 3.3V    | 10Mbps  | MAX3485 兼容替代       |
| MAX13487 | 3.3V/5V | 500kbps | 内置自动方向控制（无需DIR引脚！） |
| SN75176  | 5V      | 10Mbps  | TI 经典型号            |

## 7. RS-485 总线拓扑与多机通信
### 7.1 总线拓扑（手拉手）
RS-485 的标准接法是**菊花链（Daisy-Chain）**：
```
主机(Master)         从机1(Slave1)      从机2(Slave2)      从机3(Slave3)
┌────────┐           ┌────────┐         ┌────────┐         ┌────────┐
│  STM32 │           │  BMS1  │         │  BMS2  │         │  BMS3  │
│  MAX485├──A────────┤A  MAX  ├────A────┤A  MAX  ├────A────┤A  MAX  │
│        ├──B────────┤B  485  ├────B────┤B  485  ├────B────┤B  485  │
└────────┘           └────────┘         └────────┘         └────────┘
                                                                 │
                                                           终端电阻 120Ω
```
**不推荐星形拓扑**：总线中间不能随意分叉（T形分支会造成信号反射）。
### 7.2 多机通信的关键：地址
RS-485 只是物理层，本身没有地址概念。地址由上层协议（如 **Modbus RTU**）来实现：
```
主机发送查询帧：
[从机地址][功能码][寄存器地址][数量][CRC]
   0x05      0x03    0x0000   0x0004  xxxx
所有从机都能"听到"这帧数据，但只有地址=0x05的从机会回应。
```
本项目用的就是 **Modbus RTU** 协议来读取电池数据。
## 8. RS-485 终端电阻
### 8.1 为什么需要终端电阻
RS-485 总线在高速通信时，信号到达线缆末端会发生**反射**，反射回来的信号会叠加在后续数据上，造成误码。
终端电阻（120Ω）安装在总线两端，将信号能量吸收掉，消除反射。
```
主机端                                              末端从机
┌─────┐    A ──────────────────────────────────── A ┌─────┐
│MAX  │                                             │MAX  │
│485  │    B ──────────────────────────────────── B │485  │
└─────┘                                             └─────┘
  │                                                   │
  ├──[120Ω]──┤                               ├──[120Ω]──┤
  (主机端终端)                               (末端终端，最重要！)
```
### 8.2 什么时候加终端电阻
| 场景                     | 需要终端电阻？  |
| ---------------------- | -------- |
| 短线（<10米）、低速（<9600bps）  | 可以不加     |
| 长线（>10米）或高速（>19200bps） | **必须加**  |
| 调试阶段通信不稳定              | 先加120Ω试试 |

### 8.3 上拉/下拉偏置电阻
总线空闲时（没有设备在发送），A/B 线悬空，电平不确定，可能触发误码。  
解决方法：在主机端加偏置电阻：
- A线接 VCC 方向的上拉电阻（560Ω典型值）
- B线接 GND 方向的下拉电阻（560Ω典型值）
```
VCC
 │
[560Ω] ── A线（总线）
[560Ω] ── B线（总线）
 │
GND
```
## 9. 结合本项目理解代码
### 9.1 项目中两路电池通信的区别
```
电池1（智深）─── UART3 TTL直连 ──── STM32 PA9/PA10
  ✓ 近距离，TTL电平，无需485芯片
  ✓ 全双工，无需DIR控制
  ✓ dirPin = 0 (代码中占位，不实际使用)

电池2         ─── RS-485 ────────── MAX485 ── STM32 UART5 + DIR GPIO
  ✓ 差分信号，抗干扰强
  ✓ 半双工，需要 DIR 引脚控制收发切换
  ✓ dirPin = USART2_DIR_Pin
```
### 9.2 代码中 dirPin = 0 的含义
```c
/* modbus_task.c - 电池1，UART3 TTL直连 */
mbMasterStack.hardware.max485.dirPin = 0;           /* 无 DIR 引脚 */
mbMasterStack.hardware.max485.dirPort = GPIOA;      /* 任意有效端口指针，占位用 */
```

FreeModbus 的硬件抽象层在发送前后会调用：
```c
HAL_GPIO_WritePin(dirPort, dirPin, GPIO_PIN_SET);   // dirPin=0 → 写BSRR[0]，无效果
HAL_GPIO_WritePin(dirPort, dirPin, GPIO_PIN_RESET);
```
`dirPin = 0` 意味着 `HAL_GPIO_WritePin` 的 pin 参数为0，写操作不会影响任何GPIO，是个优雅的"空操作"设计。
### 9.3 有 DIR 引脚的一路（电池2/USART2）
```c
/* modbus_task.c - USART2，有485 DIR控制 */
mbMasterStack3.hardware.max485.dirPin = USART2_DIR_Pin;
mbMasterStack3.hardware.max485.dirPort = USART2_DIR_GPIO_Port;
```
FreeModbus 的端口层在发送前会拉高 DIR，发送完成后拉低 DIR，实现收发切换。
### 9.4 Modbus RTU 帧格式（项目实际使用）
本项目使用 Modbus RTU 读取电池寄存器（功能码 0x03）：
```
请求帧（主机→电池）：
┌─────────┬──────────┬────────────────┬───────────────┬───────┐
│ 从机地址 │  功能码  │  起始寄存器地址 │   寄存器数量   │  CRC  │
│  1Byte  │  1Byte   │    2Bytes      │    2Bytes     │ 2Byte │
│   0x04  │   0x03   │  0x00 0x00     │  0x00 0x01   │  xx   │
└─────────┴──────────┴────────────────┴───────────────┴───────┘

响应帧（电池→主机）：
┌─────────┬──────────┬──────────────┬──────────┬───────┐
│ 从机地址 │  功能码  │  字节数      │  寄存器值 │  CRC  │
│  0x04   │   0x03   │    0x02      │  xx xx   │  xx   │
└─────────┴──────────┴──────────────┴──────────┴───────┘
```
## 10. 常见问题与调试技巧
### 10.1 能发送但无法接收
**可能原因**：
1. DIR 引脚没有在发送完成后拉低（一直处于发送模式）
2. A/B 线接反（换一下试试）
3. 波特率/奇偶校验不匹配

**排查方法**：
```
1. 用万用表量 DIR GPIO，发送后是否恢复低电平？
2. 用示波器量 A、B 线，空闲时A应比B高约200mV
3. 用逻辑分析仪或USB转485模块在PC端抓包对比
```
### 10.2 通信偶尔出错（间歇性误码）
**可能原因**：
1. 缺少终端电阻
2. 总线过长或走线混乱（走线靠近强电/马达）
3. 多台设备同时发送（地址冲突或时序问题）
4. DIR 切换时序不对（发送结束前提前切回接收）
### 10.3 接了多台设备都不通
**可能原因**：
1. 从机地址设置错误（每台从机地址必须唯一）
2. 总线接线用了星形而非菊花链
3. 某台从机的A/B接反，导致总线被拉反
**快速定位**：一次只接一台从机，逐台排查。
### 10.4 USB转485工具调试方法
购买一个 **USB转RS485模块**（某宝约10-30元），配合 PC 上的串口调试助手，可以直接监听总线数据：
```
PC ─── USB转485模块 ─────┬──── A线 (总线)
                         └──── B线 (总线)
                              ↑
                         接在总线上即可旁听，不影响正常通信
```
## 11. 总结对比表
| 对比项     | UART TTL（电池1） | RS-485（电池2）    |
| ------- | ------------- | -------------- |
| 物理层     | 单端，0V/3.3V    | 差分，A-B         |
| 连接线     | TX + RX + GND | A + B（可不共GND）  |
| 最大距离    | ~1米           | ~1200米         |
| 抗干扰     | 弱             | 强              |
| 多机支持    | 不支持           | 支持（32节点）       |
| 双工方式    | 全双工           | 半双工（标准）        |
| DIR控制   | 不需要           | **必须**         |
| 需要转换芯片  | 不需要           | 需要（如MAX485）    |
| 本项目UART | UART3         | UART5          |
| 本项目DIR  | 无（dirPin=0）   | USART2_DIR_Pin |
| 典型应用    | 板间短距通信        | 工业现场总线         |
